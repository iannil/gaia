# 最佳实践与反模式

本文档汇总了使用 GAIA 框架的最佳实践和常见反模式，帮助你更高效地执行任务。

---

## Phase G 最佳实践

### ✅ 最佳实践

| 实践 | 说明 |
|------|------|
| 广度优先搜索 | 先广泛搜索现有方案，再决定是否自建 |
| 评估信誉 | 检查 Stars、维护状态、社区活跃度 |
| 理解原理 | 不只是封装工具，要理解背后的原理 |
| 交叉验证 | 综合多个专家观点，形成融合方案 |

### ❌ 反模式

| 反模式 | 问题 | 正确做法 |
|--------|------|----------|
| 直接开始实现 | 跳过搜索，重复造轮子 | 先 `search-skill` |
| 盲目使用低信誉方案 | 可能存在安全/质量问题 | 优先使用 Tier 1 来源 |
| 照搬代码 | 不理解原理直接复制 | 理解核心逻辑后实现 |
| 单一来源 | 只参考一个专家/项目 | 交叉验证多个来源 |

## Phase A 最佳实践

### ✅ 最佳实践

| 实践 | 说明 |
|------|------|
| MVP 驱动 | 明确核心 vs. 锦上添花 vs. 不做 |
| 排名优于估算 | 优先级排序 > 时间估算 |
| 依赖链排序 | 按依赖关系排执行顺序 |
| 提前识别风险 | 预判技术/知识风险并准备缓解方案 |

### ❌ 反模式

| 反模式 | 问题 | 正确做法 |
|--------|------|----------|
| "都重要" | 缺乏优先级 | 明确 P0/P1/P2/P3 |
| 过早估算 | 时间估算不准确 | 先排优先级，实现中再估算 |
| 边界模糊 | 范围蔓延 | 明确"不做"清单 |
| 忽视依赖 | 执行时发现阻塞 | 提前评估依赖关系 |

## Phase I 最佳实践

### ✅ 最佳实践

| 实践 | 说明 |
|------|------|
| 克制原则 | 只安装绝对必要的依赖 |
| 按优先级实现 | P0 → P1 → P2 → P3 |
| 立即记录 | 坑点/经验立即写入 evolution.json |
| 小步验证 | 每个功能完成后立即验证 |

### ❌ 反模式

| 反模式 | 问题 | 正确做法 |
|--------|------|----------|
| 工具囤积 | 安装"可能有用"的工具 | 只安装必需的 |
| 跳跃实现 | 不按优先级，挑容易的做 | 严格按 P0 → P1 顺序 |
| 依赖记忆 | "下次记住"而非记录 | 立即写入 evolution.json |
| 大批实现 | 一次性做很多才验证 | 每完成一个就验证 |

## Phase A (验收) 最佳实践

### ✅ 最佳实践

| 实践 | 说明 |
|------|------|
| 明确 Pass/Fail | 核心需求 100% 通过才算完成 |
| 验证证据 | 保存截图/日志/测试结果 |
| 复盘沉淀 | 成功和改进点都记录下来 |
| 文件即记忆 | 所有交付物妥善保存 |

### ❌ 反模式

| 反模式 | 问题 | 正确做法 |
|--------|------|----------|
| "基本可以" | 降低验收标准 | 严格按 MVP 清单验收 |
| 跳过复盘 | 经验没有沉淀 | 认真复盘并更新记录 |
| 代码未提交 | 交付物未持久化 | 提交到 Git，备份配置 |
| 只有成功 | 不记录改进点 | 坦诚记录待改进之处 |

## 跨阶段最佳实践

### 1. 需求驱动

| ✅ 正确 | ❌ 错误 |
|---------|---------|
| "这个功能解决了 X 问题" | "这个技术很厉害，用上" |
| "当前阻塞是 Y，优先解决" | "先把所有可能的功能做了" |

### 2. 站在巨人肩上

| ✅ 正确 | ❌ 错误 |
|---------|---------|
| "现有 Skill 完全匹配，直接用" | "自己写一个更好的" |
| "学习这个算法的核心原理" | "直接复制代码" |

### 3. 工匠与专家结合

| ✅ 正确 | ❌ 错误 |
|---------|---------|
| 规划时思考"为什么" | 规划时纠结"怎么做" |
| 实现时关注细节质量 | 实现时频繁推翻架构 |
| 两者适度切换 | 一直处于一种模式 |

### 4. 动态演化

| ✅ 正确 | ❌ 错误 |
|---------|---------|
| 每次任务后更新记录 | "经验都在脑子里" |
| 定期回顾 evolution.json | 从不回顾历史记录 |
| 将临时方案沉淀为 Skill | 每次都重新发明轮子 |

## 通用建议

1. **沟通明确**: 使用简写指令时，确保参数填写完整
2. **适度留痕**: 记录关键决策，但不过度文档化
3. **定期清理**: 不用的 Skill 及时清理，保持工具箱精简
4. **持续优化**: 每次任务后反思流程，持续改进

## 常见问题 FAQ

**Q: 是否必须严格按 G → A → I → A 顺序？**

A: 这是推荐的标准流程。对于非常简单的任务，可以适当简化，但核心思想（先思考再实现）应保留。

**Q: 什么时候可以跳过 Phase G 的搜索？**

A: 几乎没有这种情况。即使是全新领域，也应先尝试 `skill-from-masters` 寻找专家方法论。

**Q: evolution.json 应该多详细？**

A: 记录"未来会用到"的信息。包括有效参数、坑点、成功技巧。过度详细反而降低可读性。

**Q: MVP 如何界定？**

A: 问自己："没有这个功能，用户能用吗？"不能 → 核心功能。能 → 锦上添花。
