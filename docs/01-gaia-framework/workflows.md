# 工作流与指令映射

本文档定义了 GAIA 框架中的标准工作流和常用指令映射，用于简化协作中的指令调用。

---

## 简写指令映射表

### Phase G 相关

| 简写指令 | 完整指令 | 说明 |
|----------|----------|------|
| `search-skill` | "针对 [需求]，在可信源中搜索现成的 Skill" | 市场优先路径 |
| `skill-from-github` | "针对 [需求]，在 GitHub 寻找相关核心算法并学习原理" | 原理优先路径 |
| `skill-from-masters` | "针对 [需求]，启动专家方法论构建 Skill" | 方法论优先路径 |

### Skill 管理相关

| 简写指令 | 完整指令 | 说明 |
|----------|----------|------|
| `ls-skills` | "列出所有已安装的 Skills" | 查看 Skill 清单 |
| `check-skills` | "检查所有 Skills 的更新" | 更新检查 |
| `evolve` | "将本次使用 [Skill名] 的经验 '[具体经验]' 记录到 evolution.json" | 记录经验 |
| `clean-skills` | "清理本次任务引入的临时 Skills，保留 [常用 Skill 列表]" | 清理工具箱 |

### Phase A 相关

| 简写指令 | 完整指令 | 说明 |
|----------|----------|------|
| `define-mvp` | "定义本次任务的 MVP：核心功能、锦上添花、明确不做" | MVP 定义 |
| `prioritize` | "对所有子任务进行优先级排序 (P0-P3)" | 优先级排序 |
| `assess-risks` | "评估技术依赖、知识依赖和潜在风险" | 风险评估 |

### Phase I 相关

| 简写指令 | 完整指令 | 说明 |
|----------|----------|------|
| `implement-p0` | "按 P0 优先级实现 [任务]" | 实现核心功能 |
| `record-pitfall` | "将遇到的坑点 '[问题描述]' 和解决方案记录到 evolution.json" | 记录坑点 |

### Phase A (验收) 相关

| 简写指令 | 完整指令 | 说明 |
|----------|----------|------|
| `verify-mvp` | "对照 MVP 清单逐一验证，给出 Pass/Fail 结果" | 验收核心需求 |
| `retro` | "对本次任务进行复盘：成功之处、待改进点、核心洞察" | 复盘 |
| `persist` | "确保所有交付物（代码、文档、配置）已妥善保存" | 成果持久化 |

## 完整工作流示例

### 工作流：创建新功能

```
1. search-skill
   └─ 找到？→ 跳到步骤 4
   └─ 未找到 → 继续

2. skill-from-github (技术问题)
   或 skill-from-masters (方法论问题)

3. define-mvp
4. prioritize
5. assess-risks
6. implement-p0 (按优先级实现)
7. record-pitfall (遇到坑点时)
8. verify-mvp
9. retro
10. persist
11. clean-skills
```

## 自定义指令映射

根据项目需求，可以在 CLAUDE.md 中添加个人化的指令映射：

```markdown
## 自定义指令映射

| 简写 | 完整指令 | 使用场景 |
|------|----------|----------|
| `deploy-prod` | "执行生产环境部署流程" | 部署 |
| `run-tests` | "运行全部测试并生成报告" | 测试 |
```

## 指令调用建议

1. **优先使用简写**: 简写指令更简洁，减少 token 消耗
2. **明确参数**: 使用 `[参数名]` 占位符时，调用时替换为具体值
3. **组合使用**: 多个简写可以组合使用，完成复杂任务
4. **定期维护**: 根据实际使用频率调整映射表
